# Notification Policy Audit — 2026-05-01

This audit walks every outbound email site discovered via:

```
grep -rn "resend\.emails\.send\|sendEmail\|sendPaybackerEmail\|dispatchPocketAgentAlert" src/
```

Each site is classified into one of three buckets:

| Bucket          | Priority | Behaviour                                                                                              | Respects `email_digest_optout`? |
| --------------- | -------- | ------------------------------------------------------------------------------------------------------ | ------------------------------- |
| TIME-CRITICAL   | `1`      | Send immediately. Examples: account/auth, payment receipts, dispute milestones, fraud-style overcharge. Mirror to WhatsApp/Telegram if user has linked. | No                              |
| DIGESTIBLE      | `2`      | Enqueue for a daily/weekly digest. Dedupe by `dedup_key`. Examples: price-increase nudges, deal alerts, contract-end reminders, weekly money digest. | Yes                             |
| MARKETING       | `3`      | Lifecycle / nurture sequences. Honour broad marketing opt-out. Never dispatched on WhatsApp/Telegram. | Yes (and global marketing flag) |

`WhatsApp/Telegram parity` flags whether the site already mirrors via
`dispatchPocketAgentAlert` (Pocket Agent fan-out) or whether parity should be
added in PR D. PR A only lands the schema + this doc; the call sites are
unchanged.

## Migration summary

`supabase/migrations/20260502_notification_queue.sql` adds:

- `public.notification_queue` table with `id`, `user_id`, `kind`, `payload`,
  `priority` (1/2/3 check), `dedup_key`, `created_at`, `dispatched_at`,
  `digest_id`.
- Unique partial index on `(user_id, dedup_key)` while
  `dispatched_at is null and dedup_key is not null` — so the same alert can
  only sit in the queue once per user.
- Index on `(user_id, dispatched_at)` for the digest scan.
- Row-level security enabled with a single `service_role only` policy. No
  client-side reads or writes.
- `profiles.email_digest_optout boolean not null default false`. Time-critical
  and transactional emails ignore this flag.

No code paths are wired to the queue in this PR. Existing send sites continue
to behave exactly as before.
